# The backlog (the work bank)

The bank of work items the Larder arena can be asked to build. A **run** at a given **level** pulls a
subset of these IDs and hands them to agents (one agent per slice). The catalog is
[`backlog.md`](backlog.md); this file is the field spec and the rules.

## Why a bank (not a fixed feature list)

One fixed feature set gives one experiment. A *bank* lets us compose many runs at different
intensities from the same grounded, deduplicated pool — and grow it over time without re-deriving the
collision map each time. Every item is **grounded in a real arena file** (CooperBench-style natural
overlap, never adversarial booby-traps): they collide because they genuinely touch the same shared
substrate, not because we rigged them to.

## Item shape

Each entry in `backlog.md` carries:

| Field | Meaning |
|-------|---------|
| **ID** | Stable, type-prefixed, zero-padded: `F-NN` feature · `UX-NN` ux-fix · `D-NN` debt · `M-NN` migration · `IMP-NN` improvement. IDs are permanent — never renumber. |
| **Type** | `feature` (new vertical capability) · `ux-fix` (a UX/correctness gap, e.g. free-text→dropdown) · `debt` (a curated obstacle that funnels agents through shared files) · `migration` (dependency/framework churn) · `improvement` (polish, a11y, perf, coverage). |
| **Size** | `S` / `M` / `L` rough build effort. |
| **Independently implementable** | True ⇒ it passes its own tests in isolation. The bank invariant: every item is independently-implementable-but-conflicting (each works solo; pairs contend on shared code). |
| **Touch-set** | The real files it modifies. This is what computes collisions — two items collide iff their touch-sets intersect. |
| **Collides with** | The IDs whose touch-sets intersect this one's. The collision graph. |
| **Acceptance criteria** | **Behavioral** pass/fail statements (assertable against the seed), never structural. These are the raw material for the hidden gate — *test behavior, not "matches reference's classes."* |

## How collisions are read

`backlog.md` opens with a **highest-collision leaderboard** (items with the largest touch-set
overlap) and the synthesis **overlap notes** naming the chokepoints — currently `src/shared/serialize.ts`,
`src/shared/types.ts` + `src/client/api.ts`, `src/server/routes/items.ts`, the duplicated item form
across `ItemsPage.tsx` + `ItemDetailPage.tsx`, `prisma/schema.prisma`, and `package.json`. When
composing a high-density run, pick items that share a chokepoint.

## Levels consume IDs, not items directly

A [level](../levels.md) groups IDs into **slices** (one agent per slice) and stacks slices as it ramps.
The backlog stays the flat, ID-addressable pool; `levels.md` is the composition over it.

## Adding / editing items

- Hand-edit `backlog.md` freely — it is now the source of truth (not regenerated).
- New item ⇒ next free number in its type prefix. Set its touch-set, then add its ID to the
  `collides with` list of every item it shares a file with (and vice-versa).
- Re-running the review workflow proposes *candidates*; fold genuinely new ones in by hand so IDs stay
  stable.

## Provenance

`backlog.md` was generated by the `larder-backlog-review` workflow — 11 dimension reviewers reading
`arena/base` source, then a synthesis/dedup pass (79 candidates → 58 items). Regenerating is not
required and would churn IDs; treat the file as authored from here on.
